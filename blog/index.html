How to create a good TS project i.e. how to put a ton of badges in a readme

Starting up

A good project can be defined in many ways, so let me clarify what I think about when i say 'a good project'. It at least has the following items:

  • Git
  • Versioning
  • A name (preferably a relevant & memorable one)
  • Linting & formatting aka code style

There's more but we'll get to them further down. For now, let's begin by maintaining a project folder and initiate git & npm:

We will be maintaining an open source api-starter project based on Prisma.

Planning

A minima amount of thoughts should go into beginning a project:

  1. A Name
  2. Directory Structure / Organizing
  3. Libraries being used

It's always nice to have a name for a project.

The structure of a project can change a lot, but having an idea of what goes where from the beginning is nice. Especially if working in a team or having to include new people in the starting fase. The structure should of course be flexible and subject to change whenever reasonable, more so in the beginning than later on.

Some libraries are for configuration, being widely used or just makes sense for the project's scope. Therefore a list of libraries we'd like to use is great starting point.

Thinking about which libraries to use can be a bit hard when starting out, don't spend half an hour on this, just sketch the libraries you remember and deem necessary:

  • Prisma
  • Graphql-yoga
  • TypeScript
  • TSLint
  • Prettier
  • Husky
  • Lint-staged
  • Commitlint
  • Commitizen

Initialize npm

Let's begin our project by maintaining a folder and initialize a new npm package

mkdir kandelborg-api-starter && cd "$_"
# press enter to accept defaults or enter values where appropriate:
npm init

After going through the npm init questions, it says


About to write to /home/mathias/kandelborg-api-starter/package.json:

{

  "name": "kandelborg-api-starter",

  "version": "1.0.0",

  "description": "",

  "main": " ",

  "scripts": {

    "test": "echo \"Error: no test specified\" && exit 1"

  },

  "author": "Mathias Kandelborg",

  "license": "SEE LICENSE IN LICENSE"

}



Is this OK? (yes)

Press enter a last time and a package.json with that text is created. We're already getting pretty far!

Next up is adding a license before we forget - https://choosealicense.com/ is a great resource about this.

I've chosen to go with the MIT license for my open source project.

Create a file called LICENSE, without any extensions.

touch LICENSE

Input the license text, save the file and we can move on knowing people can use the upcoming source code! 😀

Now, we can start thinking about git. We don't want it to commit certain files & folders, so maintaining a gitignore is almost mandatory. There's a really handy tool called gitignore.io that'll generate a boilerpate gitignore based on the technologies defined. In this case, it's:

  • Git
  • Node

in addition to the different operating systems (because all people should be able to easily contribute to an OSS project):

  • Windows
  • Linux
  • macOS

That'll generate a quite long file, which I'm not going to share here, but here's the url.

Save that in a .gitignore and we will be on our way.

# Kandelborg API starter

This project will get you up an running with a GraphQL API in less than 5 minutes!

Initialize & use git

We've finally come to a point where we might want to start versioning our source code.

That's easily done by initialising git in the current directory:

git init

Having worked on several projects with git so far, I can vouch for how messy a git history can become. The whole thing can get complicated fast and suddenly git isn't as helpful as it could or should be.

I say "no more!" - let's be good internet citizens and lint & format everything, even our commits!

Installing commitizen

  • Install commitizen globally
  • Initialize a commit message convention adapter
npm i -g commitizen
commitizen init cz-conventional-changelog -D --save-exact

Add the following script the the script section of package.json:

If you want to, the test command in package.json can safely be removed

    "cz": "git-cz"

As a bonus, we've just unlocked our first badge!

If we want to ensure consistency in the source code, checking for X before committing is a good solution. Fortunately git already have hooks and there's a library called husky to easily utilize those hooks. It's really easy to get started, here's what we've got to do:

Lint staged files

  • Install lint-staged
  • Create a lint-staged config file
npm i -D lint-staged
touch .lint-stagedrc

Since we're going to use prettier for code style, we can preemptively use it in lint-staged:

{
  "*.{json,css,md}": ["prettier --write", "git add"],
  "*.{ts,tsx}": ["prettier-tslint fix", "git add"]
}

Enforce linted commits

  • Install commitlint
  • Create a commitlint config file
npm install -D @commitlint/cli @commitlint/config-conventional
touch commitlint.config.js

With this content:

module.exports = {extends: ['@commitlint/config-conventional']}

Do things on every commit

  • Install husky
  • Create a husky config file
npm i -D husky
touch .huskyrc.js
  • Run lint-staged on pre-commit and commitlint on each commit message:
const tasks = (arr) => arr.join(' && ')

module.exports = {
  hooks: {
    'pre-commit': tasks(['lint-staged']),
    'commit-msg': tasks(['commitlint -E HUSKY_GIT_PARAMS'])
  }
}

Format files

Note: I always format my files on save - your editor probably has a setting that allows actions on save

Most editors have a plugin or extension system that can utilize a formatter or on built in

First off, let's get an edítorconfig file:

EditorConfig helps maintain consistent coding styles for multiple developers working on the same project across various editors and IDEs.

— Editor config homepage

touch .editorconfig

And some standard settings:

root = true

[*]
indent_style = space
indent_size = 2
charset = utf-8
trim_trailing_whitespace = true
insert_final_newline = true

Consistent formatting with prettier

In order to consistently enforce formatting, using something like Prettier or StandardJS is (in my book) a must. Let's get started with Prettier (and pretter-tslint for TSLint):

npm i -g prettier
npm i -D prettier-tslint

Create a prettier config file

touch .prettierrc.js

And fill it with your configuration of choice, here's an example:

module.exports = {
  tabWidth: 2,
  printWidth: 80,
  semi: false,
  trailingComma: 'none',
  singleQuote: true,
  bracketSpacing: true,
  jsxBracketSameLine: true,
  arrowParens: 'always'
}
